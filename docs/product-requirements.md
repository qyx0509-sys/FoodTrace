# 《食藏录》产品需求文档（MVP）

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 食藏录 |
| 产品形态 | 微信小程序、Flutter Android/iOS App、共用后端 |
| 当前阶段 | 第一阶段：需求与技术设计 |
| 文档状态 | 待审核 |
| 产品定位 | 私人美食打卡与店铺管理工具 |

## 2. 产品愿景

“食藏录”帮助用户把分散在聊天、地图收藏、相册和记忆中的餐厅信息，整理成只属于自己的美食档案。产品强调真实、私密和可持续记录，不建设公开点评社区，也不替用户形成面向公众的商家排名。

## 3. 产品目标

### 3.1 MVP 目标

1. 用户可以通过微信身份登录，并在小程序与 App 间识别为同一用户。
2. 用户可以从腾讯位置服务搜索店铺，也可以手动录入店铺。
3. 用户可以创建、查看、编辑和删除自己的美食记录。
4. 用户可以用“想去”“已打卡”“黑名单”管理三类默认清单。
5. 用户可以记录评分、人均消费、菜品、备注、标签和图片。
6. 用户可以通过列表和地图检索自己的记录，并查看基础统计。
7. 所有用户数据必须按 `userId` 隔离，任何用户均不能读取或修改其他用户的数据。

### 3.2 成功信号

MVP 不设置公开增长指标，先用以下产品可用性信号判断是否达标：

- 用户能在 2 分钟内完成首次店铺记录。
- 已授权用户能在小程序和 App 中看到同一份个人数据。
- 记录在列表、详情、地图和统计中的结果一致。
- 弱网或短时断网时，用户仍能查看最近缓存，并保留未提交的编辑草稿。
- 越权访问、永久密钥泄露和跨用户缓存污染的自动化测试均为零失败。

## 4. 用户与使用场景

### 4.1 目标用户

MVP 仅包含一个角色：`User`。用户管理自己的店铺与美食记录，不存在商家、运营人员、好友或公开访客角色。

### 4.2 核心场景

1. 看到别人推荐一家店，将它加入“想去”。
2. 到店用餐后，把“想去”更新为“已打卡”，补充评分、消费、菜品、备注和图片。
3. 对体验不佳的店铺标记“黑名单”，留下避雷信息。
4. 出门前按距离、标签、评分或状态查找自己的店铺记录。
5. 在地图上回顾去过的店，以及附近尚未打卡的店。
6. 查看自己的记录数量、状态分布、平均评分和常用标签。

## 5. 产品原则与边界

### 5.1 必须遵守

- 所有业务数据均为私人数据，默认不可公开、不可分享。
- 商家不能认领店铺，也不能修改、回复或删除用户记录。
- 不提供全站评分、商家排行榜、公开评论流或用户关注关系。
- 服务端从认证上下文取得 `userId`；客户端不得指定或代替其他 `userId`。
- 客户端只保存可公开的应用标识、短期令牌和短期上传凭证，不保存微信、地图 WebService 或对象存储永久密钥。
- 敏感配置全部通过环境变量或部署平台的密钥管理服务注入。
- MVP 优先采用简单、可测试、可回滚的实现，不提前建设微服务、消息队列和管理后台。

### 5.2 MVP 明确不做

- 公开动态社区、公开点评与榜单；
- 商家入驻、商家后台与商家申诉；
- 优惠券、支付、会员和商业化；
- 好友、关注、共享清单与多人协作；
- 复杂年度报告；
- AI 推荐；
- 运营管理后台；
- 自定义清单和一次到店对应多条用餐日志。

## 6. 核心概念与业务规则

### 6.1 `Store`

`Store` 是用户记录中的店铺快照，来源为腾讯 POI 或手动录入。即使两个用户选择同一个腾讯 POI，服务端也分别保存各自的 `Store`，避免用户修改互相影响。

MVP 规则：

- 必填：`name`、`latitude`、`longitude`。
- 坐标统一为 `GCJ-02`，地图展示和附近计算不得混用坐标系。
- 腾讯 POI 保存 `providerPoiId` 和必要的地址快照；外部 POI 数据变化不会覆盖用户已保存的数据。
- 手动店铺允许编辑；腾讯 POI 店铺允许用户修改个人快照，但不回写腾讯平台。
- 同一用户、同一 `providerPoiId` 只保存一份有效店铺快照。

### 6.2 `FoodRecord`

`FoodRecord` 是用户针对一家店维护的当前私人记录。MVP 中同一用户、同一家店最多一条记录，重复用餐通过编辑原记录反映当前判断。多次到店历史是后续候选能力。

记录状态 `status`：

| 标识 | 中文 | 说明 |
| --- | --- | --- |
| `WANT_TO_GO` | 想去 | 尚未到店或暂未形成体验 |
| `VISITED` | 已打卡 | 已完成至少一次到店 |
| `BLACKLISTED` | 黑名单 | 用户不希望再次选择的店铺 |

状态可以双向切换，不保留公开可见的状态变更历史。切换为 `WANT_TO_GO` 时，历史体验字段默认保留并二次确认，避免误删；用户可以主动清空。

### 6.3 评分

- 字段为 `overallRating`、`tasteRating`、`environmentRating`、`serviceRating`。
- 合法范围为 1.0 至 5.0，步长为 0.5。
- 四项评分彼此独立，`overallRating` 不由其他三项自动计算。
- 所有评分在数据库层面允许为空；`WANT_TO_GO` 页面默认不展示评分输入。
- 未评分记录不参与平均分统计。

### 6.4 消费与日期

- `perCapitaPrice` 表示人均消费，必须大于或等于 0，MVP 币种固定为 `CNY`。
- `visitedAt` 为用户本地日期，可为空；创建“已打卡”记录时默认当天。
- 统计按用户设置的 `timezone` 解释日期，默认 `Asia/Shanghai`。

### 6.5 菜品、标签与备注

- 菜品分为 `RECOMMENDED`（推荐菜）和 `AVOIDED`（踩雷菜）。
- 同一记录同一分类下的菜名去重，单个名称 1 至 50 个字符。
- 标签为用户私有，可复用；标签名去除首尾空格后按大小写不敏感去重。
- 单条记录最多 10 个标签；单个标签名 1 至 20 个字符。
- `notes` 最多 5,000 个字符，内容仅作为纯文本处理。

### 6.6 图片

- 单条记录最多 9 张图片，支持 `JPEG`、`PNG`、`WebP`。
- 单张原图最大 10 MB；客户端可压缩，但服务端必须再次校验元数据。
- COS Bucket 必须为私有读；数据库保存 `objectKey`，读取时返回短时签名 URL。
- 上传使用服务端签发的临时、最小权限凭证，路径限定在当前 `userId` 与 `recordId` 下。
- 图片可排序和删除；删除记录后其图片不再可访问，并进入对象清理流程。

### 6.7 默认清单

MVP 的三类清单是由 `FoodRecord.status` 生成的系统视图，不创建独立 `lists` 表，也不允许改名、删除或共享：

- 想去：`status = WANT_TO_GO`
- 已打卡：`status = VISITED`
- 黑名单：`status = BLACKLISTED`

## 7. 功能需求

### 7.1 登录与账号

- `FR-AUTH-01`：微信小程序使用 `wx.login` 获取临时 `code`，由后端换取微信身份。
- `FR-AUTH-02`：Android/iOS App 使用微信开放平台授权，由后端完成授权码交换。
- `FR-AUTH-03`：后端签发短期 `accessToken` 和可轮换的 `refreshToken`。
- `FR-AUTH-04`：刷新令牌仅以哈希形式存储，退出登录后当前会话失效。
- `FR-AUTH-05`：同一微信开放平台下优先使用 `unionId` 关联小程序与 App；无法取得 `unionId` 时不得猜测或自动合并账号。
- `FR-AUTH-06`：本地缓存按认证用户命名空间保存，退出后清除令牌和私人缓存。

### 7.2 店铺发现与录入

- `FR-STORE-01`：按关键词和当前城市搜索腾讯 POI。
- `FR-STORE-02`：可使用地图中心点或当前位置辅助搜索。
- `FR-STORE-03`：搜索结果显示名称、分类、地址和距离。
- `FR-STORE-04`：选择 POI 后保存标准化快照，不直接信任客户端提交的店铺详情。
- `FR-STORE-05`：无法搜索到的店铺可以手动填写名称、地址、坐标、分类和电话。
- `FR-STORE-06`：仅在用户明确提交后持久化店铺，不保存普通搜索历史到服务端。

### 7.3 记录管理

- `FR-RECORD-01`：创建记录时必须选择或创建店铺，并选择三种状态之一。
- `FR-RECORD-02`：可填写四项评分、人均消费、日期、两类菜品、备注、标签和图片。
- `FR-RECORD-03`：详情页显示店铺信息与完整私人记录。
- `FR-RECORD-04`：用户可以编辑自己的店铺快照和记录。
- `FR-RECORD-05`：删除前必须二次确认；删除后记录立即从列表、地图和统计中消失。
- `FR-RECORD-06`：并发编辑使用 `version` 做乐观锁，冲突时提示用户刷新后重试。

### 7.4 搜索、筛选与排序

- `FR-QUERY-01`：可按店铺名、地址、菜名、标签和备注关键词搜索自己的记录。
- `FR-QUERY-02`：可按状态、标签、评分区间、消费区间和到店日期筛选。
- `FR-QUERY-03`：可按最近更新、最近到店、评分、人均消费和距离排序。
- `FR-QUERY-04`：距离排序只有在用户授权位置且记录存在坐标时可用。
- `FR-QUERY-05`：筛选条件可重置；MVP 不做跨设备保存的自定义筛选器。

### 7.5 地图

- `FR-MAP-01`：在腾讯地图上标记当前用户的有效记录。
- `FR-MAP-02`：不同状态使用可辨识的标记样式，并提供图例。
- `FR-MAP-03`：移动或缩放地图后按可视区域加载标记，避免一次返回全部记录。
- `FR-MAP-04`：点击标记显示店铺摘要，可进入详情。
- `FR-MAP-05`：拒绝定位授权时仍可按城市或已有记录范围浏览地图。

### 7.6 图片上传

- `FR-IMAGE-01`：客户端向后端申请限定对象路径、大小、类型和有效期的临时上传凭证。
- `FR-IMAGE-02`：客户端直传 COS，完成后调用后端确认接口。
- `FR-IMAGE-03`：只有确认成功的对象才出现在记录中。
- `FR-IMAGE-04`：图片读取 URL 短时有效，日志不得打印签名参数或临时密钥。

### 7.7 本地缓存

- `FR-CACHE-01`：缓存最近一次列表结果、详情、统计和必要的店铺缩略信息。
- `FR-CACHE-02`：编辑页面自动保存本地草稿，提交成功后删除草稿。
- `FR-CACHE-03`：离线时允许只读最近缓存和继续编辑草稿，不支持离线创建后的自动同步队列。
- `FR-CACHE-04`：缓存包含 `schemaVersion`、`userId` 和更新时间，版本不兼容时安全失效。
- `FR-CACHE-05`：令牌存入各平台安全存储；业务缓存不得包含微信授权码或 COS 临时密钥。

### 7.8 基础统计

- `FR-STATS-01`：显示有效记录总数和三种状态数量。
- `FR-STATS-02`：显示已评分记录的平均总体评分。
- `FR-STATS-03`：显示已填写消费记录的平均人均消费。
- `FR-STATS-04`：显示使用次数最多的标签，最多 10 个。
- `FR-STATS-05`：所有统计只计算当前用户数据，不建设跨用户指标。

## 8. 非功能需求

### 8.1 安全与隐私

- 所有受保护接口必须验证 JWT，并从 JWT subject 映射 `userId`。
- 对不存在和不属于当前用户的资源统一返回 `404`，减少资源枚举信息泄露。
- 日志对令牌、微信 `code`、手机号、签名 URL 和密钥脱敏。
- 接口启用 DTO 白名单、类型转换和未知字段拒绝。
- 登录、刷新、POI 搜索和上传凭证接口需要速率限制。
- 数据库查询必须包含 `userId` 条件；跨用户访问需要集成测试覆盖。
- 生产环境仅允许 HTTPS，数据库与 COS 使用最小权限账号。

### 8.2 性能目标

在正常网络和单区域部署条件下，以服务端 P95 为口径：

- 普通读取接口小于 500 ms；
- 普通写入接口小于 800 ms；
- POI 搜索不含外部供应商超时的目标小于 2 s；
- 首屏列表默认返回不超过 20 条；
- 地图单次最多返回 500 个轻量标记，超出时提示缩小范围。

以上为 MVP 工程目标，不构成公开 SLA；上线前使用真实环境压测校准。

### 8.3 可用性与可观测性

- 所有外部请求设置超时、有限重试和明确错误提示。
- 每个请求生成 `requestId`，错误响应与结构化日志可关联。
- 健康检查区分进程存活与数据库可用状态。
- 数据库必须有自动备份与恢复演练方案；具体 RPO/RTO 在部署阶段确认。

### 8.4 兼容性与无障碍

- 小程序支持当前微信开发者工具稳定基础库及主流在用版本，初始化阶段锁定最低版本。
- Flutter 支持范围在初始化阶段依据当前稳定 SDK 和地图 SDK 要求锁定。
- 关键文本支持系统字体缩放，状态不能只依靠颜色区分，点击目标保持足够尺寸。

## 9. 埋点与隐私最小化

MVP 不接入第三方行为分析平台。服务端只保留故障排查所需的结构化运行日志，不记录用户备注、菜名全文和图片 URL 查询参数。若后续增加分析能力，必须先明确数据目的、保留周期和用户告知方式。

## 10. MVP 验收标准

### 10.1 功能验收

| 编号 | 验收场景 | 通过标准 |
| --- | --- | --- |
| `AC-01` | 微信登录 | 小程序和 App 均可完成授权、刷新令牌和退出；满足关联条件时得到同一 `userId` |
| `AC-02` | POI 搜索 | 输入关键词后返回腾讯 POI；选择结果可创建私人店铺快照 |
| `AC-03` | 手动店铺 | 可用名称、地址和地图选点创建店铺，坐标按 `GCJ-02` 保存 |
| `AC-04` | 三类记录 | 三种状态均可创建、查看和切换，且只出现在对应默认清单中 |
| `AC-05` | 体验字段 | 四项评分、消费、日期、推荐菜、踩雷菜、备注和标签可保存并正确回显 |
| `AC-06` | 图片 | 最多上传 9 张合规图片，可排序、查看和删除，客户端无永久 COS 密钥 |
| `AC-07` | 查询 | 关键词、状态、标签和数值区间筛选正确；五种排序结果稳定 |
| `AC-08` | 地图 | 当前可视区域显示当前用户标记，状态样式可辨，标记可进入详情 |
| `AC-09` | 编辑冲突 | 旧 `version` 提交不会静默覆盖新数据，并返回可处理的冲突错误 |
| `AC-10` | 删除 | 二次确认后记录从列表、详情、地图和统计消失，图片不再可访问 |
| `AC-11` | 缓存 | 断网可查看最近缓存并保留草稿；切换或退出用户后不会看到前一用户数据 |
| `AC-12` | 统计 | 总数、状态分布、平均总体评分、平均消费和常用标签与数据库样本一致 |

### 10.2 安全与质量验收

| 编号 | 通过标准 |
| --- | --- |
| `AC-SEC-01` | 用户 A 使用自己的令牌无法读取、修改、删除用户 B 的任意资源，接口返回 `404` |
| `AC-SEC-02` | 客户端产物和仓库历史中不存在微信 `appSecret`、COS 永久密钥、地图 WebService Key、JWT Secret 或数据库密码 |
| `AC-SEC-03` | 上传凭证仅能写入当前用户与记录的限定前缀，过期后不可继续使用 |
| `AC-SEC-04` | DTO 非法范围、非法类型、超长文本、未知字段和伪造 `userId` 均被拒绝 |
| `AC-QUAL-01` | 后端 lint、类型检查、单元测试、集成测试和构建全部通过 |
| `AC-QUAL-02` | 小程序 lint、TypeScript 检查、单元测试和构建全部通过 |
| `AC-QUAL-03` | Flutter analyze、单元/Widget 测试及 Android/iOS 对应构建检查全部通过 |
| `AC-QUAL-04` | Swagger 与实际 DTO、状态码和认证要求一致，关键接口有契约测试 |
| `AC-QUAL-05` | 测试结果来自实际命令输出；失败项、跳过项和未执行项均在交付说明中如实列出 |

## 11. MVP 后候选方向

- 多次用餐日志与记录时间线；
- 自定义私人清单；
- 数据导入、导出和用户自助彻底删除；
- 更丰富的个人年度回顾；
- 在明确隐私边界后的端侧智能整理能力。

这些方向不进入当前排期，需重新评审需求、数据模型与隐私影响。
