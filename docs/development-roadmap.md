# 《食藏录》开发路线

## 当前基线（2026-07-15）

已完成并纳入检查：

- 品牌统一为“食藏录 / FoodTrace”；
- pnpm workspace、NestJS、Prisma/PostgreSQL、微信 TypeScript 与 Flutter 工程；
- 环境变量边界、严格启动校验、Git 敏感文件规则；
- User/Store/FoodRecord/Tag/RecordTag/RecordImage/DishRecord 与身份/会话模型；
- 微信小程序登录、JWT access/refresh、轮换、退出与用户隔离 Guard；
- 腾讯 POI 搜索、手动店铺、记录创建/列表/详情/编辑/软删除核心 API；
- 小程序登录—搜索/建店—记录—列表—详情—今日手账的主流程；
- Flutter 真实健康检查骨架；
- 格式、Lint、类型、测试、迁移与构建 CI。

## P0 收尾

1. 在实际微信测试号配置 AppID/AppSecret 与业务域名，完成登录/POI 真机验收。
2. 在 Node 22 与 PostgreSQL 17 的干净环境重放 CI，固化首个可部署版本。
3. 使用用户 A/B E2E 数据继续覆盖更新、删除和幂等并发，而不仅是服务单测与数据库约束测试。

## P1

1. COS 私有图片：压缩、MIME/大小校验、短期凭证、进度、签名读取与孤儿清理。
2. Flutter 微信移动端登录与记录主流程；从 OpenAPI 生成或共享稳定契约。
3. 收藏/想去/再吃关系梳理、地图与基础统计、标签独立管理。
4. 今日手账模板配置化并扩展为票据、可爱手账、城市杂志三种模板。
5. 隐私政策、账号注销、数据导出与生产网关分布式限流。

## P2（本轮不做）

高级统计、年度报告、AI 推荐/生图、复杂动画、好友/共享、评论/关注、商家合作、优惠券、会员与支付。

## 每阶段质量门槛

Node/小程序执行 `pnpm check`；Flutter 执行格式、分析与测试。数据库变更必须新增迁移、验证空库重放、重复 seed 与用户隔离，不修改已发布历史迁移。任何外部服务未配置时返回清晰错误，不提供假数据或假成功。
